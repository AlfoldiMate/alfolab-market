# CLI → zsh completion: generation reference

This is the deep methodology behind the `autocomplete-cli` and
`autocomplete-cli-refresh` skills. Read it in full before generating or
refreshing a completion. The goal is a `_<cli>` completion function that:

- completes **global options** with descriptions, value enums, and file/dir args;
- dispatches into **each subcommand** and completes that subcommand's *own*
  options (not just the global ones);
- groups subcommands into **git-style labelled categories** so `fzf-tab`
  renders them as sections.

The running example throughout is a [commander.js](https://github.com/tj/commander.js)
CLI (Claude Code's own `claude` is one), but the **Adapting to other CLIs**
section covers clap / cobra / click / argparse.

---

## 0. Pick the completion directory (fpath)

zsh loads completion functions named `_<cmd>` from any directory on `$fpath`.

1. Inspect the user's fpath: `zsh -ic 'print -l $fpath' | grep -E '\.zfunc|\.zsh/completions|site-functions'`.
2. Prefer an existing user-writable dir already on fpath (commonly `~/.zfunc`).
3. If none exists, create `~/.zfunc` and ensure the rc file adds it. Check the
   rc first — many setups already have `fpath+=~/.zfunc` even if the dir is
   missing. If you must add it, insert **before** the `compinit` call:
   ```zsh
   fpath+=~/.zfunc
   ```
4. The completion file is `<fpath-dir>/_<cli>` and its first line MUST be
   `#compdef <cli>`.

> Heads-up on cached compinit: many configs run `compinit -C` (fast, skips
> scanning fpath) when `~/.zcompdump` is fresh. A **new** `_<cli>` file won't be
> discovered until the dump is rebuilt. After writing a new file, run
> `rm -f ~/.zcompdump*` and tell the user to restart the shell (or `exec zsh`).
> Editing an *existing* `_<cli>` is picked up automatically (autoload re-reads
> the file), but clearing the dump is harmless and keeps things predictable.

## 1. Introspect the CLI

### 1a. Confirm it exists and capture version
```bash
which <cli>; <cli> --version 2>/dev/null
```

### 1b. Top-level help → global options + commands
```bash
<cli> --help 2>&1
```
From the `Options:` block, record for each flag:
- long form (`--model`) and short form if any (`-m`);
- whether it takes an argument — a `<arg>` / `<arg...>` after the flag means it
  does; `[arg]` means optional argument; bare flag means none;
- whether it's **repeatable** — `<arg...>` (ellipsis) → mark repeatable;
- any **fixed value set** — commander prints `(choices: "a", "b", "c")`; also
  infer enums from prose like "Transport type (stdio, sse, http)".

From the `Commands:` block, record each subcommand name + one-line description.
Commander aliases show as `install|i` → support both.

### 1c. Per-subcommand help → that subcommand's LOCAL options

Commander **inherits all global options into every subcommand's `--help`**, so a
naive scrape would re-list 50+ globals under each subcommand. Isolate the
*local* options by diffing flag-name sets. This bash snippet does it reliably:

```bash
# Extract just the --flag names from a help text. The ${=1} word-split matters.
optnames() {
  <cli> ${=1} --help 2>/dev/null \
    | grep -oE '^  (-[a-zA-Z], )?--[a-z][a-zA-Z0-9-]+' \
    | grep -oE '\-\-[a-z][a-zA-Z0-9-]+' | sort -u
}
optnames "" > /tmp/root_opts.txt                      # global flags

# Local options of a subcommand = its flags minus the global ones:
local_opts() {
  comm -23 <(optnames "$1") /tmp/root_opts.txt
}
local_opts "mcp add"        # -> --callback-port --client-id --env --header --scope --transport ...
```
(Run this in **bash**, not zsh — `<(...)` process substitution inside a zsh
function body can collide with arithmetic parsing. Or write to temp files.)

Then re-read the subcommand help to grab the full line for each local flag
(short alias, arg type, value enum, description). Recurse one more level for
nested command groups (e.g. `plugin marketplace add`).

### 1d. Detect value enums

| Source in help | Completion value spec |
|---|---|
| `(choices: "text", "json")` | `(text json)` |
| prose "scope (local, user, project)" | `(local user project)` |
| `<dir>` / `<directory...>` / "directories" | `_files -/` |
| `<file>` / `<path>` "Path to ..." | `_files` |
| `<port>` `<amount>` `<uuid>` free value | leave action empty (`:port:`) |
| a known dynamic set (e.g. model names) | a custom helper function (see §2d) |

## 2. Generate the `_<cli>` function

Use `_arguments` for option/positional parsing and `_describe` for grouped
candidate lists. Skeleton:

```zsh
#compdef <cli>
# zsh completion for <cli> — generated by autocomplete-cli.

_<cli>() {
  local curcontext="$curcontext" state line
  typeset -A opt_args

  local -a global_opts
  global_opts=(
    '--model[Model for the session]:model:(opus sonnet haiku)'
    '(-p --print)'{-p,--print}'[Print response and exit]'
    '--output-format[Output format]:format:(text json stream-json)'
    # ... one line per global flag ...
  )

  _arguments -C \
    $global_opts \
    '1: :->command' \
    '*:: :->args'

  case $state in
    command)
      # grouped, git-style — see §3
      ;;
    args)
      case $line[1] in
        <sub1>) _<cli>_<sub1> ;;
        # ...
        *) _arguments $global_opts ;;
      esac
      ;;
  esac
}
```

### 2a. Option spec syntax (the bits you'll reuse constantly)
- `'--flag[description]'` — boolean flag.
- `'--flag[description]:label:action'` — flag takes one arg; `action` is
  `(a b c)` for a fixed set, `_files` / `_files -/`, a function name, or empty.
- `'--flag[description]::label:action'` — **optional** arg (double colon).
- `'(-s --scope)'{-s,--scope}'[desc]:scope:(local user project)'` — short+long
  pair that excludes each other.
- `'*'{-e,--env}'[desc]:env:'` — leading `*` = **repeatable**.
- `'(--bg --background)'{--bg,--background}'[desc]'` — two long aliases.
- Negation/exclusion groups go in the leading `( ... )`.

### 2b. Positionals
- `'1:name:'` first positional (no completion action), `'2:json:'` second, etc.
- `'1:target:(stable latest)'` positional with a fixed set.
- `'*:::args:'` remaining args.

### 2c. Per-subcommand handler
One function per command group. Offer the subcommand's **local options +
`--help`** (NOT the full global set — the session/global flags are irrelevant
inside `mcp add` and would drown the list). Example:

```zsh
_<cli>_mcp() {
  local curcontext="$curcontext" state line
  local -a add_cmds manage_cmds
  add_cmds=( 'add:Add a server' 'add-json:Add via JSON' )
  manage_cmds=( 'get:Show a server' 'list:List servers' 'remove:Remove a server' )
  _arguments -C '1: :->cmd' '*:: :->args'
  case $state in
    cmd)
      _describe -t mcp-add-commands    'add servers'      add_cmds
      _describe -t mcp-manage-commands 'inspect & manage' manage_cmds
      ;;
    args)
      case $line[1] in
        add)
          _arguments \
            '*'{-e,--env}'[Set environment variables]:env:' \
            '(-s --scope)'{-s,--scope}'[Configuration scope]:scope:(local user project)' \
            '(-t --transport)'{-t,--transport}'[Transport type]:transport:(stdio sse http)' \
            '1:name:' '2:commandOrUrl:' \
            '(-h --help)'{-h,--help}'[Show help]'
          ;;
        *) _arguments '(-h --help)'{-h,--help}'[Show help]' ;;
      esac
      ;;
  esac
}
```

### 2d. Dynamic value helpers
For a value set that's a known list but appears in several places (e.g. model
aliases), define a small helper and reference it as the action:
```zsh
_<cli>_models() { local -a m; m=(opus sonnet haiku); _describe -t models 'model' m }
# ...used as:  '--model[...]:model:_<cli>_models'
```

## 3. Categorize commands (git-style groups) — the headline feature

Instead of ONE flat `_describe`, emit **several** `_describe` calls, each with a
distinct `-t <tag>` and a human group header. zsh renders one
`[header]` section per call (via the user's
`zstyle ':completion:*:descriptions' format '[%d]'`), and `fzf-tab` shows them
as separate sections — exactly like git's "main porcelain / ancillary" split.

```zsh
command)
  _describe -t auth-commands  'authentication'      auth_cmds
  _describe -t ext-commands   'extensions & config' ext_cmds
  _describe -t maint-commands 'maintenance'         maint_cmds
  ;;
```

**Choosing categories** (do this thoughtfully per CLI — it's the part that makes
the completion feel hand-crafted):
- Group by *user intent / mental model*, not alphabetically. Typical buckets:
  authentication, configuration/extensions, build/run, inspect/query,
  lifecycle (install/enable/update), authoring/release, maintenance, advanced.
- 2–6 groups at a level is the sweet spot. Don't over-split.
- Put the most-used group first (groups render in call order; users can also
  pin order with `zstyle ':completion:*' group-order ...`).
- Tiny command sets (2–3 items) can stay a single group — still give it a header.
- Keep `help` quietly in an inspect/manage group.

Same technique applies at every level: top-level commands, and the subcommand
lists inside each group function.

## 4. Activate

```bash
zsh -n <fpath-dir>/_<cli>      # syntax check — MUST pass
rm -f ~/.zcompdump*            # force compinit to rediscover/rebuild
```
Tell the user to open a new terminal or run `exec zsh`.

## 5. Verify it actually completes (zpty harness)

Don't just trust the syntax check — drive real completion in a pseudo-terminal.
zpty tests are timing-sensitive; gate on an `INIT_DONE` sentinel so the probe
never races the shell's startup echo. Save as a script and run with `zsh -f`:

```zsh
probe() {                                   # $1 = the line to complete (trailing space matters)
  zmodload zsh/zpty
  export TERM=xterm COLUMNS=200 LINES=60
  zpty Z 'zsh -f'
  zpty -w Z 'fpath=(<FPATH-DIR> $fpath); autoload -Uz compinit; compinit -u; setopt nobeep; zstyle ":completion:*" menu no; zstyle ":completion:*:descriptions" format "[%d]"; PROMPT="R%% "'
  zpty -w Z 'print INIT_DONE'
  local acc="" c
  while true; do zpty -r -t Z c 2>/dev/null && acc+="$c" || sleep 0.1; [[ "$acc" == *INIT_DONE* ]] && break; done
  sleep 0.3; while zpty -r -t Z c 2>/dev/null; do :; done
  zpty -w -n Z "$1"; zpty -w -n Z $'\t'
  local out="" i=0
  while (( i++ < 25 )); do zpty -r -t Z c 2>/dev/null && out+="$c" || sleep 0.1; done
  print -r -- "### '$1<TAB>'"
  print -r -- "$out" | tr -d '\r' | sed $'s/\x1b\\[[0-9;?]*[a-zA-Z]//g' | grep -v '^[[:space:]]*$' | grep -vi 'INIT_DONE\|R%%' | head -30
  zpty -d Z
}
probe "<cli> "                 # expect the grouped [headers] + commands
probe "<cli> <sub> --"         # expect that subcommand's local options
probe "<cli> <sub> --scope "   # expect the value enum
```
Run each probe in its own fresh process if back-to-back sessions get noisy.

## 6. Adapting to other CLIs (not commander.js)

The introspection in §1 assumes commander's help layout. Adjust the scrape:

- **clap (Rust)** — many clap CLIs ship `<cli> completions zsh` or
  `clap_complete`. **Check `<cli> completion[s] --help` first** and just use the
  native generator if present. Otherwise: subcommands appear under
  `Commands:` / `SUBCOMMANDS:`, options under `Options:`; clap does NOT inherit
  global options into subcommands, so skip the diff — scrape each subcommand's
  options directly. Enums show as `[possible values: a, b]`.
- **cobra (Go)** — usually ships `<cli> completion zsh`. Prefer it. Else: help
  has `Available Commands:` and `Flags:` + `Global Flags:` (already separated —
  use `Flags:` as local, `Global Flags:` as global). 
- **click / argparse (Python)** — no inheritance; `Commands:` block lists
  subcommands; options under `Options:`. argparse enums show as `{a,b,c}` in the
  usage line. click may offer `_<CLI>_COMPLETE=zsh_source <cli>`.
- **Always prefer a native `completion`/`completions` generator** if the CLI has
  one — only hand-roll when it doesn't, or when the native one lacks the
  categorized groups / value enums the user wants (in which case you can still
  hand-roll for the richer UX).

When in doubt, read 2–3 subcommand `--help` outputs and confirm whether global
flags repeat under subcommands — that single fact decides whether you diff
(commander) or scrape directly (everyone else).
